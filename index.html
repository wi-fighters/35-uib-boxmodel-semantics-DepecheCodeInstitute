This page is for my practice using valid semantic elements, with a particular nod to a11y. It is important to use built-in HTML elements where possible, as they generally offer better code structure and transparency, and often come with cool built-in features like native keyboard accessibility. For more on web accessibility, see also a11yproject).

THIS IS AN ARTICLE ELEMENT. I added a heading because html validator suggested it.

The <article> element identifies a self-contained piece of content which could theoretically be distributed to other websites and platforms as a stand-alone unit. The <article> element is a good choice to contain entire blog posts, news articles, and similar content... [while it is] functionally similar to using <div> elements, and the display and styling work the same. However, using the <article> element instead of <div> provides more semantic information to screen readers, search engines, and third-party applications.(HTML Article Element: Learn When (And Where) To Use It Now)
  • buttons should be given an accessible name in the <button> element that concisely describes the button's functionality for assistive technologies to access when they parse the document and generate an accessibility tree. Assistive technology then uses the accessibility tree to navigate and manipulate page content (MDN; See <button>: The Button element for more information).
  • native buttons operate with the Enter key and the Spacebar key, which carries through to javaScript onclick events.
  • to make a button keyboard operable all you need is the onclick event and it works with Enter and Spacebar keys (will give this a try after I learn how to combine js with my html/css)
  • <div role=”button”> controls don’t work with the Enter AND Spacebar keys or they don’t work with the keyboard at all. They’re either skipped right over when tabbing or pressing enter, or spacebar keys have no effect because they only have a mouse-dependent onclick JavaScript event.
  • do not use a button for a link, as they have separate respective button and link roles.
  • The reality is that most faux <div> buttons are not keyboard or screen reader accessible at all, and as a developer you’re responsible for implementing the full functionality of the control you use based on its role. So either use native buttons or mimic them completely (Paul J. Adam).
  • See Accessible ARIA buttons for more information.

THIS IS AN ASIDE ELEMENT.
the <aside> element is used to identify content that is related to the primary content of the webpage, but does not constitute the primary content of the page. Author information, related links, related content, and advertisements are exampes[sic] of content that may be found in an aside element (HTML5 Aside Element: Here Are The Good (And Not So Good) Uses For It).

CSS Guidelines I referred to in making this website, included:

  • Smashing Magazine's Typographic Design Survey (Michael Martin), which taught me to consider the contrast between serif headings and sans-serif body text; outlined the most popular typefaces and drew my attention to Helvetica and Lucinda Grande?; suggested a heading:body font-size ratio of 2:1, or to use the Fibonacci sequence for font increases; and suggested a line-height of 1.5em
  • Web Designer Depot's 5 Rules for Choosing the Perfect Web Typeface (Ben Moss), which suggested that I use the text 1Illinois to test typeface choice; to look for typefaces with large and open counters(i.e. the white space in the centre of letters like o and c)
  • pick newer typefaces that are more likely to have more legible characters at both smaller and larger font-sizes
